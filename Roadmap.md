## 📦 Детализация спринтов

### Спринт 1: Foundation & Client Bot
**Цель:** Заложить архитектурный фундамент и реализовать навигацию клиента по каталогу услуг.

* **Архитектура:** Утверждение схемы БД и развертывание базовой инфраструктуры (VPS, Docker, CI/CD).
* **База данных:** Создание таблиц Users, Specialists, Services, Slots, Appointments.
* **Авторизация:** Обработка команды `/start`, сохранение Telegram ID и запроса номера телефона клиента.
* **Каталог:** Реализация Inline-меню для выбора услуги и специалиста (с поддержкой пагинации).
* **Календарь:** Генерация и отображение доступных временных слотов на ближайшие 30 дней.

### Спринт 2: Booking Engine & Admin Panel
**Цель:** Реализовать самую критичную часть системы (бронирование без конфликтов) и дать администратору инструменты контроля.

* **Бронирование (Critical Path):** Реализация транзакционного создания записи с защитой от двойного бронирования (пессимистичные блокировки на уровне БД).
* **Управление клиентом:** Функция отмены записи клиентом с проверкой правила «не позднее 3 дней».
* **Доступ администратора:** Проверка прав доступа для команды `/admin`.
* **Управление каталогом:** CRUD-операции для администратора (добавление/редактирование специалистов, услуг, настройка рабочих часов).
* **Управление расписанием:** Просмотр записей (фильтрация по дате/специалисту), перенос и отмена записей администратором.

### Спринт 3: Notifications, QA & Launch
**Цель:** Завершить бизнес-логику фоновыми процессами, протестировать систему и вывести продукт на рынок.

* **Планировщик:** Интеграция фонового воркера (Celery или APScheduler) для отложенных задач.
* **Триггеры:** Настройка автоматических Telegram-уведомлений клиенту при изменении статуса записи администратором.
* **Напоминания:** Настройка автоматической отправки сообщений за 24 часа до визита.
* **Тестирование (QA):** Проведение нагрузочного тестирования на конкурентный доступ к одному слоту.
* **Приемка (UAT):** Демонстрация полного цикла заказчику, исправление найденных багов (Feature Freeze).
* **Запуск (Go-Live):** Деплой финальной версии в Production-окружение, подключение мониторинга ошибок (Sentry).

## 🚩 Вехи и точки контроля (MCP)

| Веха (Milestone) | Описание | Критерий успеха |
| :--- | :--- | :--- |
| **M1: Схема БД утверждена** | Фиксация структур данных. | ER-диаграмма согласована, транзакционная модель подтверждена. |
| **M2: MVP Клиентского бота** | Завершен Спринт 1. | Можно пройти по меню до выбора времени без ошибок (Mock-данные). |
| **M3: Core Booking Ready** | Готово ядро транзакций. | Успешно пройден тест на одновременную запись (дабл-букинг исключен). |
| **M4: Feature Freeze** | Завершен Спринт 2 и 3. | Разработка нового функционала остановлена. Фокус только на багфиксе. |
| **M5: UAT Sign-off** | Пользовательское тестирование. | Заказчик подписал акт приемки, критичных багов нет. |
| **M6: Production Launch** | Бот доступен для реальных клиентов. | Мониторинг активен, сделана первая реальная запись на проде. |

## ⚠️ Управление ключевыми рисками

* **Гонка данных (Двойная запись):** Блокируется на уровне БД (`SELECT FOR UPDATE`). Проверяется обязательным авто-тестом в Спринте 2.
* **Часовые пояса:** Все даты и время хранятся строго в UTC. Конвертация в Московское время (UTC+3) происходит только на слое UI (отрисовка сообщений).
* **Сбои уведомлений:** Использование планировщика с функцией авто-рестарта и логированием недоставленных сообщений (блокировка бота пользователем).
